# RFC-0002: Collection Domain Model

**Status:** Proposed
**Author:** Project contributors
**Date:** 2026-07-16

---

## Summary

TWW3 Companion organises a user's knowledge around a **Workspace**. A Workspace owns a shared Mod Library and one or more Collections. Mods are defined once in the library; Collections reference them through membership records that hold collection-specific organisation and rationale.

The model is source-neutral, preserves unresolved knowledge, distinguishes user-authored information from imported metadata and derived results, and records game compatibility as versioned observations rather than a single mutable status.

This RFC defines domain meaning, ownership, lifecycle rules, and invariants. It does not choose a storage engine, serialisation format, UI framework, or import grammar.

---

## Problem

A collection-only model duplicates knowledge when the same mod appears in several curated lists. It also blurs facts about a mod with the reasons a player includes it in a particular campaign.

The domain model must support:

- one reusable record for each known mod;
- several independently organised Collections;
- external identities without making Steam Workshop the domain root;
- missing dependencies and compatibility targets that are useful precisely because they are missing;
- evidence that changes across game patches;
- imports that never silently overwrite player-authored knowledge;
- future Profiles without duplicating Mods or Collection membership data.

---

## Chosen Approach

The Workspace is the aggregate root.

```text
Workspace
├── target game version (optional)
├── Mod Library
│   ├── Mods
│   ├── Relationships
│   └── Game Compatibility Observations
└── Collections
    ├── Collection Memberships
    └── Profiles (future behaviour deferred)
```

This approach keeps reusable knowledge in one place while allowing each Collection to organise the same Mod differently.

---

## Domain Model

### Workspace

The user-owned root of the application domain. It contains the shared Mod Library and all Collections. It may declare an optional target game version used when assessing compatibility evidence.

Workspace is a domain boundary, not a decision about whether data is stored in one file, several files, or a database.

### Mod

A source-neutral library record representing a modification as understood by the companion. It does not represent installed files or a single publishing page.

A Mod owns:

- a stable internal identity;
- a preferred display name and optional aliases;
- zero or more Source References;
- optional imported source metadata;
- shared user-authored notes;
- shared relationships and game compatibility observations.

The internal identifier's concrete format is deferred to storage design.

### Source Reference

An external location or identity through which a Mod is published or obtained. Steam Workshop is the first supported source, but the domain permits future local, GitHub, or other references.

A Source Reference is unique within a Workspace by the combination of source type and external identifier. One Mod may have several Source References; the same Source Reference may not identify two Mods.

### Collection

A named, curated set of Mods with collection-level notes. A Collection owns Collection Membership records rather than copies of Mods.

### Collection Membership

The link between one Collection and one library Mod. It owns context that may differ between Collections:

- optional category;
- collection-local tags;
- rationale and notes;
- user-declared tracking state;
- optional ordering or load-order notes.

Ordering information is documented knowledge. TWW3 Companion does not enforce load order.

A Mod may appear at most once in a Collection and may appear in any number of different Collections.

### Profile

A Profile belongs to exactly one Collection and will later describe a playable subset or configuration of that Collection. RFC-0002 defines only this ownership boundary. Profile fields, activation rules, and overrides are deferred to the v0.3 design.

### Relationship Record

A shared knowledge record describing a Dependency or Compatibility Claim.

A relationship target may be:

- a resolved Mod in the library; or
- an unresolved external reference containing enough identity or descriptive information to resolve later.

Resolving a target updates the reference without replacing the relationship or its evidence.

### Dependency

A directional relationship from one Mod to a target. Its type is one of:

- **requires** — necessary for intended operation;
- **recommends** — strongly suggested but not required;
- **optional integration** — enables additional behaviour when present.

### Compatibility Claim

A claim about whether two Mods work together. Its state is one of:

- **compatible**;
- **incompatible**;
- **patch required**;
- **unknown / needs verification**.

Absence of a claim means no knowledge has been recorded. It is distinct from an explicit unknown claim.

Compatibility Claims may be scoped to a game version or version range through their evidence.

### Evidence

Evidence explains why a relationship or compatibility observation is believed. It may record:

- a source reference or URL;
- observation date;
- game version or patch label;
- Mod/source version where known;
- free-form notes;
- provenance, such as user observation, mod author statement, or imported/community source.

Evidence is encouraged but not mandatory. The product must distinguish sourced claims from unsourced notes.

### Game Compatibility Observation

A versioned observation attached to a Mod. It records:

- game version or patch label;
- result: **works**, **works with caveats**, **broken**, or **unverified**;
- observation date;
- optional Mod/source version;
- notes and Evidence;
- provenance.

Observations are retained as history. **Last known good game version** and **latest known broken game version** are derived summaries, not manually authoritative fields.

Conflicting observations remain visible with their provenance. They are not silently reconciled or overwritten.

---

## Information Ownership

The model distinguishes three classes of information.

### User-authored

Examples include preferred display name, shared notes, Collection rationale, categories, tags, ordering notes, and personal observations. Imports must not silently overwrite these values.

### Imported source metadata

Examples include source title, author or publisher, source URL, description summary, and last-observed source details. Values retain source and observation time where practical. Imports may enrich blank fields automatically; conflicts with user-authored values require an explicit choice.

### Derived

Examples include duplicate warnings, missing-dependency findings, last-known-good summaries, health findings, counts, and search indexes. Derived data is reproducible from authoritative records and must not become its sole authoritative copy.

---

## Lifecycle Rules

### Collection removal

Removing a Mod from one Collection deletes only that Collection Membership. It does not alter the shared Mod or memberships in other Collections.

### Library deletion

Deleting a Mod from the shared library is permitted only through an explicit confirmed operation.

Before confirmation, the application must:

1. list every Collection containing the Mod;
2. state that the affected memberships and their categories, tags, rationale, tracking state, and ordering notes will be deleted;
3. require explicit confirmation.

On confirmation, the Mod, all of its Collection Memberships, and relationships it owns as the source Mod are deleted. Incoming relationship records whose target is the deleted Mod are preserved and their targets become unresolved.

An ordinary Collection-removal action must never trigger library deletion.

---

## Core Operations

The domain supports intention-revealing operations:

- add or update a library Mod;
- attach or resolve a Source Reference;
- create, rename, or delete a Collection;
- add, edit, or remove a Collection Membership;
- create or update a relationship or compatibility observation;
- resolve an external relationship target;
- delete a library Mod through confirmed cascade deletion.

Arbitrary mutation that bypasses validation is not part of the public domain contract.

---

## Import Boundary

Imports are staged:

```text
Parse external input
→ normalise candidate identities
→ compare with library and collection
→ produce an import preview
→ resolve conflicts
→ validate proposed changes
→ apply atomically
```

The preview reports additions, matches, duplicates, unresolved entries, warnings, and conflicts before changing the Workspace. Failed validation or interrupted persistence must leave the existing Workspace unchanged.

Detailed Markdown grammar and identity matching rules belong to the import RFC.

---

## Validation Model

Validation outcomes have three severities:

- **Error** — the change cannot be applied, such as assigning one external source identity to two Mods.
- **Warning** — the change may proceed after review, such as retaining an unresolved dependency.
- **Information** — a useful result requiring no intervention, such as an imported item already existing in a Collection.

Failures must identify the affected entity and a proposed remedy. Silent dropping, automatic destructive merging, and unexplained last-write-wins behaviour are forbidden.

---

## Invariants

1. A Source Reference identifies at most one Mod within a Workspace.
2. A Collection contains a given Mod at most once.
3. A Profile belongs to exactly one Collection.
4. Collection-specific organisation lives on Collection Membership, not Mod.
5. Unresolved relationship targets are valid stored knowledge.
6. User-authored information is not silently overwritten by imports.
7. Derived values are reproducible from authoritative records.
8. Library deletion requires a complete impact warning and explicit confirmation.
9. Relationship evidence survives target resolution and deletion of the target Mod.
10. Game compatibility history is retained; derived latest-status summaries do not replace observations.

---

## Verification Criteria

Later domain tests must demonstrate that:

- one Mod can belong to several Collections without duplicating shared notes;
- memberships retain independent category, tags, rationale, and tracking state;
- duplicate Steam Workshop identities cannot create two Mods;
- unresolved dependencies survive persistence/export and can later be resolved;
- removing one membership does not alter the Mod or other Collections;
- confirmed library deletion removes all memberships after reporting every affected Collection and preserves incoming relationships as unresolved;
- imported metadata cannot silently replace user-authored values;
- derived health and compatibility summaries can be discarded and recreated;
- a Profile cannot belong to more than one Collection;
- observations `works on 7.2 → broken on 7.3 → works on 7.4` remain in history and derive 7.4 as last known good and 7.3 as latest known broken.

---

## Alternatives Considered

### Independent Collection documents

Each Collection could own complete Mod records while a registry attempts to reconcile shared identities. This makes individual Collection export simple but duplicates knowledge and introduces synchronisation ambiguity. Rejected in favour of one shared library.

### Knowledge graph

All concepts could be represented as generic nodes and edges. This is expressive but adds abstraction and validation cost before the project needs graph-level flexibility. Rejected as premature complexity.

### Steam Workshop as canonical identity

Using Workshop IDs as domain identity simplifies the first importer but makes the model dependent on one publishing platform and cannot naturally represent local or mirrored Mods. Rejected in favour of internal identity plus Source References.

---

## Deferred Decisions

This RFC does not decide:

- concrete internal identifier format;
- JSON, SQLite, or other persistence schema;
- filesystem locations and atomic save mechanism;
- Markdown grammar and detailed import matching;
- UI layout or confirmation-dialog presentation;
- health-score formula;
- Profile fields and activation behaviour;
- automated Steam metadata retrieval.

These decisions belong to later v0.0.2 architecture work.

---

## Consequences

- Existing documentation that calls Collection the top-level unit must be updated to introduce Workspace.
- Importers must distinguish library matching from Collection membership creation.
- Collection export will eventually need rules for packaging referenced shared knowledge into a portable snapshot.
- Deletion and relationship resolution require explicit domain operations.
- Compatibility assessment can be transparent because it is based on retained, versioned observations.
