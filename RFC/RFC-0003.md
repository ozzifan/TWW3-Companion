# RFC-0003: Storage Architecture

**Status:** Proposed
**Author:** Project contributors
**Date:** 2026-07-16

---

## Summary

TWW3 Companion stores each Workspace in one local SQLite database. SQLite is embedded in the application distribution and requires no separate database installation, service, account, or administration.

SQLite is the canonical working store. A versioned, lossless JSON format provides inspection, backup, sharing, and migration between tools; direct JSON editing is not a primary workflow.

The initial release targets Windows and provides both a one-step installer and a portable ZIP. Both distributions include the application runtime and SQLite support.

---

## Problem

The accepted domain model in [RFC-0002](RFC-0002.md) requires storage that can:

- preserve one shared Mod Library across several Collections;
- enforce unique Source References and Collection Memberships;
- retain unresolved relationship targets, evidence, and compatibility history;
- apply confirmed imports without partial changes;
- distinguish authoritative records from rebuildable derived data;
- migrate as the model evolves;
- remain offline-first and under the user's control;
- ship as a self-contained Windows application;
- support backup, restore, and portable export.

A storage failure must not silently lose, partially apply, or overwrite user-authored knowledge.

---

## Chosen Approach

### Canonical store

Each Workspace is stored in one SQLite database file. The file is the canonical working representation of that Workspace.

SQLite is an embedded library, not an external service. The application bundle includes the required runtime and SQLite driver. Users do not install SQLite separately, configure a server, create database credentials, or use command-line tools.

### Portable representation

TWW3 Companion provides a documented, versioned, lossless JSON export. JSON is used for:

- user-controlled backup;
- inspection outside the application;
- sharing and archival;
- restore into a new Workspace;
- migration to other tools.

JSON is not the live canonical store, and direct editing is not a primary product requirement.

### Platform and distribution

Version 0.1 targets Windows only. Other desktop platforms are deferred.

Two Windows distributions are supported:

1. a conventional one-step installer as the default;
2. a portable ZIP that runs without installation or administrator access.

Both distributions must work on a clean supported Windows environment without separately installed runtimes or SQLite tools.

---

## Storage Boundary

Domain and application code access persistence through a storage boundary. SQL and SQLite-specific types do not cross into the domain or UI.

The storage layer has four responsibilities:

### Workspace repository

Loads and persists authoritative Workspace records while preserving the ownership boundaries and invariants defined by RFC-0002.

### Transaction boundary

Applies each confirmed domain operation or resolved import as one atomic transaction.

### Migration runner

Inspects the schema version and upgrades supported older databases before they are opened for editing.

### Export and restore service

Produces and validates the lossless JSON representation without exposing rebuildable caches as authoritative data.

---

## Identity

Workspace, Mod, Collection, Collection Membership, Profile, Relationship, Evidence, and Game Compatibility Observation records use stable UUID identifiers.

UUIDs are:

- generated locally without a network dependency;
- independent of database row order and external publishing services;
- preserved unchanged through export, restore, and migration;
- treated as opaque identifiers by users and application features.

Steam Workshop IDs and future external identities remain Source Reference fields. They are not internal record identifiers.

The exact UUID version and binary or textual database encoding are implementation choices, provided exported identifiers use one documented canonical textual representation and remain stable.

---

## Relational Structure

The SQLite schema is normalised around the accepted domain entities rather than storing the entire Workspace as an opaque JSON blob.

Authoritative tables represent:

- Workspace metadata;
- Mods and aliases;
- Source References and imported source metadata;
- Collections and Collection Memberships;
- collection-local categories and tags;
- Relationships and unresolved targets;
- Evidence;
- Game Compatibility Observations;
- future Profiles once their domain design is accepted.

Foreign keys express ownership and reference boundaries. Database constraints reinforce, but do not replace, domain validation.

At minimum, storage constraints enforce:

- unique UUID identity for every record;
- uniqueness of source type plus external identifier within a Workspace;
- at most one membership for a given Mod in a Collection;
- exactly one owning Collection for each Profile;
- valid references between owned records;
- required schema and export format versions.

User-authored, imported, and derived information remain distinguishable in storage. Imported values retain source and observation time where required by RFC-0002.

---

## Derived Data

Search indexes, counts, duplicate findings, health findings, and compatibility summaries may be stored for performance.

Derived data:

- is never the sole authoritative copy of knowledge;
- can be invalidated in the same transaction as authoritative changes;
- can be discarded and rebuilt;
- is excluded from lossless JSON unless a future export specification marks a value as informational only.

Failure to rebuild a cache must not corrupt or invalidate authoritative Workspace data.

---

## Write and Import Transactions

Every persisted mutation follows this sequence:

```text
Domain operation
→ domain validation
→ begin SQLite transaction
→ persist authoritative changes
→ update or invalidate derived data
→ commit
```

If any step fails, the transaction rolls back and the previous Workspace state remains authoritative.

Import parsing, identity matching, preview, and conflict resolution do not mutate the Workspace. Only a fully resolved and confirmed import enters the transaction boundary. The complete confirmed import is committed or rolled back as one unit.

Arbitrary writes that bypass domain operations and validation are not part of the public application contract.

---

## Single-Instance Behaviour

Only one TWW3 Companion process may run for the current Windows user.

On launch, the application acquires an operating-system-level application lock before loading settings or opening a Workspace. If another TWW3 Companion process already owns the lock, the new process:

1. displays an error stating that TWW3 Companion is already running;
2. does not open or modify settings or Workspace data;
3. exits.

The rule applies across installed and portable copies. The lock must be released automatically by Windows when the owning process exits or crashes. A persistent lock file is not authoritative.

This application-wide rule replaces per-Workspace multi-process editing and read-only secondary instances.

---

## Opening a Workspace

Opening follows this sequence:

```text
Identify file
→ inspect file and schema version
→ reject unsupported or newer schemas
→ create a pre-migration backup when required
→ migrate inside a transaction
→ validate required invariants
→ expose the Workspace to the application
```

Invalid, corrupt, inaccessible, or unsupported files are not modified. Errors identify the affected file, explain why it could not be opened, and offer an appropriate next action where one exists.

---

## Schema Versioning and Migration

The database contains an explicit schema version. Migrations are ordered, forward-only, and bundled with the application.

Before the first migration performed by a newer application version, TWW3 Companion creates a timestamped backup using a SQLite-safe backup operation. Copying an actively open database file directly is not an approved backup mechanism.

All changes for one migration run occur inside a transaction. If migration or post-migration validation fails:

- the transaction rolls back;
- the pre-migration database remains usable by the previous application version;
- the backup is retained;
- the application reports the failed migration and does not open the Workspace for editing.

An older application must reject a database whose schema version is newer than it supports. It must not attempt a downgrade or silently open the file.

Migration support may have a documented minimum source version. If a database is older than that minimum, the application must require migration through a supported intermediate release or restoration from a supported export; it must not guess at a conversion.

---

## SQLite Recovery

SQLite transaction and recovery mechanisms provide crash safety. The application does not implement its own partial-save repair format.

The chosen SQLite journal mode is an implementation detail, but packaging, backup, and recovery tests must account for any temporary journal or write-ahead-log files it creates. A Workspace is considered safely backed up only when created through an approved SQLite backup or export operation.

---

## JSON Export

Every export contains:

- an export format identifier and version;
- the Workspace UUID and metadata;
- all authoritative domain records and stable UUIDs;
- Source References and imported metadata;
- Collection Membership organisation and rationale;
- resolved and unresolved relationships;
- Evidence and provenance;
- complete Game Compatibility Observation history.

Exports exclude rebuildable search indexes, caches, counts, and derived summaries.

The format uses deterministic field names and ordering where practical so exports are suitable for inspection and source-control diffs. The detailed JSON schema will be versioned under `schemas/` before implementation.

Exporting never mutates the Workspace.

### Collection snapshot export

The lossless JSON format defined by this RFC represents a complete Workspace. Exporting one Collection as a portable snapshot is a separate requirement identified by RFC-0002.

A future import/export design must define how a Collection snapshot packages:

- the selected Collection and its Memberships;
- every referenced shared Mod;
- relevant Source References, Relationships, Evidence, and compatibility history;
- unresolved targets needed to preserve the Collection's recorded knowledge;
- provenance identifying the source Workspace without exposing unrelated Collections.

RFC-0003 neither rejects nor specifies scoped Collection export. Until that design is accepted, the application must not describe a full-Workspace export as a privacy-preserving way to share one Collection.

---

## Restore

Restore follows this sequence:

```text
Parse export
→ verify format and version
→ validate all records and identities
→ build a new temporary Workspace database
→ validate database invariants
→ atomically place or replace the destination
```

Invalid exports do not create or partially replace a Workspace.

Replacing an existing Workspace requires:

1. a summary of the destination being replaced;
2. an automatic backup of the existing destination;
3. explicit user confirmation.

Restore preserves stable UUIDs. It does not merge into an existing Workspace; merge behaviour belongs to the import architecture.

## Filesystem Locations

### Installed distribution

Application binaries are installed through the Windows installer. Per-user settings, default Workspace files, and managed backups live in TWW3 Companion's managed application-data directories under the current Windows user profile.

Users may explicitly create or save Workspace files elsewhere.

### Portable distribution

The portable ZIP stores application settings beside the executable in its portable folder. Users choose where Workspace database files are created or opened.

The portable application does not write settings to the installed application's managed directories and does not write to Steam or game directories.

### User-selected locations

The application may open Workspace files in user-selected local or synchronised folders, but it guarantees correctness only through its own single-process and transactional behaviour. The application-wide lock protects one Windows user on one machine; it cannot prevent another machine from opening a copy of the same database.

Users must be warned not to open or edit a live Workspace concurrently on multiple machines through OneDrive, Dropbox, or similar synchronisation. Concurrent access through a synchronised folder is unsupported and may cause conflicting copies or database corruption. Lossless JSON export is the supported transfer mechanism between machines until a separate synchronisation design is accepted.

Behaviour of third-party synchronisation, backup, antivirus, or filesystem tools remains outside the storage contract. The application must report access and locking failures without modifying the Workspace.

---

## Uninstall and Data Retention

The installed application's uninstaller removes application files by default and leaves user data intact.

The uninstaller offers an unchecked option:

> Remove TWW3 Companion settings, managed Workspaces, and managed backups.

If selected, the uninstaller:

- displays the managed application-data location that will be removed;
- requires confirmation;
- removes only TWW3 Companion settings and data inside its managed application-data directories;
- reports deletion failures.

The uninstaller never searches the disk for Workspace files. Workspaces, exports, or backups saved elsewhere remain untouched.

The portable distribution has no uninstaller. Deleting its folder may remove portable settings and files deliberately stored inside that folder. TWW3 Companion never removes portable Workspace files stored elsewhere.

---

## Error Model

Storage failures use the validation severities defined by RFC-0002 where applicable and provide:

- the affected Workspace or file;
- the operation that failed;
- whether any change was committed;
- a specific remedy or safe next action where available.

The application must not:

- silently create a replacement for a corrupt Workspace;
- continue after a failed migration;
- partially apply a confirmed import;
- overwrite a newer schema;
- delete user-selected external files during uninstall;
- present a cache-rebuild failure as authoritative data loss.

Diagnostic details may be written to a local log, but user-authored notes, imported descriptions, and other Workspace content must not be included unless required and clearly disclosed.

---

## Verification Criteria

Storage and packaging tests must demonstrate that:

- a Workspace round-trips through save and reload without losing authoritative data;
- database constraints reinforce every storage-level invariant in RFC-0002;
- each domain operation commits completely or leaves the previous state unchanged;
- a confirmed import commits or rolls back as one unit;
- unresolved targets and Evidence survive save, reload, export, and restore;
- deleting a target Mod preserves incoming relationships as unresolved;
- derived data can be discarded and rebuilt;
- every supported old schema migrates without authoritative data loss;
- a failed migration preserves the original and its backup;
- an older application rejects a newer schema;
- JSON export and restore produce equivalent authoritative Workspaces;
- corrupt, invalid, locked, and inaccessible files produce actionable errors without modification;
- a second process explains that TWW3 Companion is already running and exits before accessing data;
- installer and portable packages run on clean supported Windows environments without separate runtimes or SQLite tools;
- default uninstall preserves user data;
- selected data removal deletes only managed application-data directories;
- portable removal leaves Workspace files outside its folder untouched.

---

## Alternatives Considered

### Canonical single JSON file

A single JSON file is directly inspectable and easy to copy. It requires rewriting the Workspace for ordinary changes, places transaction and uniqueness enforcement entirely in application code, and complicates crash recovery as the model grows. Rejected in favour of embedded SQLite plus lossless JSON export.

### Directory of JSON documents

Splitting records across files reduces individual rewrite size and improves textual diffs. It creates cross-file consistency, partial-save, identity, and backup problems that duplicate database responsibilities. Rejected for canonical storage.

### External database service

A server database can provide strong concurrency and administration features. It conflicts with offline-first operation, self-contained distribution, and the single-user desktop scope. Rejected.

---

## Deferred Decisions

This RFC does not decide:

- programming language, UI framework, or SQLite library;
- exact table and column names;
- exact UUID version or database encoding;
- database filename extension;
- detailed JSON Schema fields;
- backup retention count and cleanup policy, which the storage implementation plan must define before application code lands;
- installer technology;
- code-signing and update distribution;
- import grammar, matching, and merge behaviour;
- cloud synchronisation or collaboration.

These choices require later implementation planning or separate architecture approval where they affect public behaviour.

---

## Consequences

- v0.1 can persist the accepted Workspace domain without a separate database installation.
- Storage and packaging must be tested as one product surface.
- JSON export, rather than direct database editing, fulfils portability and inspection requirements.
- Database migrations and lossless export versions become permanent compatibility responsibilities.
- Only one application process may run per Windows user.
- Uninstall behaviour must distinguish managed data from user-selected external files.
- Import architecture can rely on one atomic persistence boundary after preview and conflict resolution.
- Acceptance requires README and roadmap updates that make Windows the initial platform target and identify other desktop platforms as deferred.
