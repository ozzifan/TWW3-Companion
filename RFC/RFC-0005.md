# RFC-0005: Initial UI Architecture

**Status:** Proposed
**Author:** Project contributors
**Date:** 2026-07-17

---

## Summary

TWW3 Companion uses C#/.NET with Avalonia and MVVM for its initial Windows desktop interface. The application has one main window, starts on a Home screen without opening a Workspace, and uses a persistent Workspace sidebar with master and detail regions after a Workspace opens.

Version 0.1 exposes only workflows required by its milestone. Shared Mod knowledge and Collection-specific Membership knowledge remain visibly separate, editing is explicit, imports retain their full preview while blocking choices are resolved, and destructive actions explain their complete impact before confirmation.

The UI follows the Windows theme by default, supports explicit System, Light, and Dark choices, and treats keyboard operation, focus, text scaling, high contrast, and assistive-technology announcements as functional requirements.

---

## Problem

The accepted domain model contains ownership boundaries that can be easy to blur in a conventional mod-list interface:

- one Workspace owns a shared Mod Library and several Collections;
- a Mod has shared knowledge;
- a Collection Membership has Collection-local organisation and rationale;
- removing a Membership is not the same as deleting a library Mod;
- import requires preview, resolution, and atomic confirmation;
- storage errors must not leave users unsure whether data changed.

The initial UI must make those distinctions understandable while remaining efficient for large lists, keyboard-friendly, offline-first, and suitable for self-contained Windows distribution.

---

## Scope

RFC-0005 defines:

- the UI technology and presentation pattern;
- main-window and navigation structure;
- v0.1 screen inventory;
- selection, editing, and draft behaviour;
- import workspace structure;
- destructive confirmation and feedback hierarchy;
- keyboard, accessibility, theme, and sizing requirements;
- UI verification boundaries.

It does not define:

- final visual branding or bespoke illustration;
- detailed v0.2 search, tag, category-management, or note-discovery UI;
- Profile workflows;
- relationship and health dashboards;
- single-Collection export UI;
- automatic update UX;
- mobile, web, macOS, or Linux interfaces.

---

## Chosen Technology

### C# and .NET

The initial application is written in C# on a supported .NET long-term-support release. The exact SDK and runtime version are pinned in the implementation plan and repository configuration before application code lands.

The installer and portable ZIP publish the application self-contained. Users do not install .NET or Avalonia separately.

### Avalonia

Avalonia provides the desktop presentation framework.

The application uses:

- the desktop target;
- standard Avalonia controls wherever they satisfy the interaction;
- UI Automation and framework accessibility properties;
- the Avalonia Fluent theme;
- framework keyboard navigation and command binding.

Custom-drawn interactive controls require a documented accessibility reason and equivalent automation behaviour.

### Platform baseline

Version 0.1 supports Windows 10 or later on x64.

ARM64 packages and non-Windows platforms are deferred. Choosing Avalonia preserves an architectural path to later desktop platforms but does not promise or schedule them.

---

## Presentation Pattern

The UI uses Model-View-ViewModel (MVVM).

### Views

Views define layout, visual states, bindings, and platform presentation. Views do not access SQLite, repositories, or domain mutation APIs directly.

### ViewModels

ViewModels own:

- screen and selection state;
- commands;
- editable drafts;
- validation messages;
- progress and cancellation state;
- accessible status announcements.

ViewModels do not execute SQL or bypass application services.

### Application services

Application services expose intention-revealing operations for:

- Workspace creation, opening, migration, backup, and restore;
- Mod and Collection Membership editing;
- import parsing, preview, resolution, planning, and application;
- destructive impact queries and confirmed deletion;
- settings and recent-Workspace management.

Services return typed results that identify success, validation failure, recoverable failure, cancellation, and blocking failure.

### Navigation state

The application shell owns navigation state. Modal decisions return typed outcomes to their caller rather than mutating global state.

---

## Process and Window Model

RFC-0003's application-wide single-instance lock is acquired before the UI loads settings or Workspace data.

If another TWW3 Companion process is running for the current Windows user, the new process displays a native blocking error explaining the reason and exits.

Version 0.1 uses one main application window.

It does not provide:

- detachable panes;
- multiple Workspace windows;
- separate Mod-detail windows;
- system-tray operation;
- background operation after the main window closes.

File pickers and blocking decisions may use owned dialogs.

---

## Startup Home

The application starts on Home and does not automatically open the most recent Workspace.

Home provides:

- Create Workspace;
- Open Workspace;
- Import into a new Workspace;
- recent Workspaces;
- missing or inaccessible recent-file indicators;
- access to basic application settings.

Selecting a recent Workspace begins the same validated open flow as Open Workspace.

Home does not inspect or migrate a Workspace until the user chooses it.

Returning Home closes the active Workspace only after any unsaved draft or in-progress operation is resolved safely.

---

## Workspace Shell

After a Workspace opens, the main window uses three persistent regions.

### Workspace sidebar

The sidebar contains:

- Mod Library;
- the Workspace's Collections;
- Import;
- application and Workspace settings where appropriate.

Only destinations implemented for the current milestone appear. Later features are not shown as disabled buttons, empty pages, or promotional placeholders.

### Master list

The centre region displays the selected Mod Library or Collection contents.

Selection updates detail state but never changes domain data.

The v0.1 list supports the information needed to identify and open entries. Search, advanced filters, and organisation tooling remain scoped to their roadmap milestones.

### Detail pane

Selecting a Mod in the library shows shared Mod detail.

Selecting a Collection Membership shows two tabs:

1. **Collection Details** — Membership category, tags, rationale, notes, tracking state, and ordering knowledge;
2. **Shared Mod** — display name, aliases, Source References, imported metadata, shared notes, Relationships, and compatibility observations where supported by the current milestone.

Collection Details is selected first for a Membership. Each tab retains visible context naming the selected Mod and Collection.

The tabs use separate ViewModels, drafts, Save commands, and validation so one action cannot save fields owned by the other record.

---

## Milestone-Bounded Screen Inventory

Version 0.1 defines and exposes:

- Home and recent Workspaces;
- Create Workspace;
- Open Workspace;
- Workspace shell;
- Mod Library list and shared Mod detail;
- Collection list and Membership detail;
- informal Markdown and Workshop-ID import;
- import preview, resolution, and confirmation;
- Workspace JSON backup and restore;
- basic settings;
- storage, migration, corruption, and single-instance messages.

The shell may accommodate future destinations structurally, but v0.1 does not display:

- global or advanced search;
- dedicated category or tag management;
- Profiles;
- health dashboards;
- full relationship editors;
- scoped Collection export.

---

## Explicit Editing

Details are read-only by default.

Editing follows:

```text
Read-only detail
→ Edit
→ local ViewModel draft
→ inline validation
→ Save
→ application-service command
→ atomic domain/storage operation
→ success announcement
→ read-only detail
```

Save is available only when the draft is valid and differs from the persisted value.

Cancel discards the draft after confirmation only when discarding would lose changes the user made.

Navigating away with an unsaved draft offers:

- Save;
- Discard;
- Stay.

Save failure retains the draft, keeps the editor open, and states that no changes were committed.

The UI does not autosave field changes.

---

## Import Workspace

Import is a full-page task inside the Workspace shell, not a modal dialog.

The workflow is:

```text
Choose source
→ full preview table
→ Needs Attention resolution queue
→ final operation summary
→ explicit Apply
→ atomic import
→ result announcement
```

### Preview table

The full RFC-0004 preview remains visible and can be grouped or filtered by Additions, Enrichments, Existing, Suggested Matches, Conflicts, Warnings, and Skipped candidates.

### Needs Attention queue

A secondary queue identifies every blocking identity choice, missing display name, ambiguous category, and unresolved conflict.

Resolve Next moves through blocking items without hiding the complete preview. Resolving an item updates the table and counts immediately.

### Confirmation

Apply remains unavailable until every blocking choice and Error is resolved and the domain plan validates.

The confirmation view shows exact operation counts and the final list of planned changes. It describes the import as additive and never as replace or synchronise.

Cancelling or leaving import prompts before discarding resolution choices. No resumable import session is persisted in v0.1.

---

## Destructive Actions

### Remove from Collection

Removing a Membership uses a compact confirmation that:

- names the Mod and Collection;
- lists Collection-local information that will be removed;
- states that the shared Mod and other Memberships remain.

### Delete from Mod Library

Library deletion uses a dedicated impact dialog.

Before enabling its destructive button, the UI loads and displays:

- every affected Collection;
- Membership category, tags, rationale, tracking state, and ordering knowledge that will be deleted;
- Relationships owned by the Mod as source that will be deleted;
- incoming Relationships that will remain with unresolved targets.

The explicit destructive button is sufficient after review. Typing the Mod name is not required.

If complete impact data cannot be loaded, deletion remains unavailable.

### Delete Collection

Collection deletion requires a separate impact summary that states Memberships will be removed while shared Mods remain. It must not invoke library deletion.

---

## Feedback and Error Hierarchy

### Field validation

Field validation appears beside the affected control. When several fields fail, the editor also provides an accessible summary linked to each field.

### Recoverable operation failure

Recoverable failures use a persistent page-level banner. The banner identifies:

- the affected object or file;
- the operation that failed;
- whether any change committed;
- a safe next action.

### Success

Successful saves, imports, backups, and restores receive a brief visible and assistive-technology announcement that does not steal focus.

### Blocking dialogs

Dialogs are reserved for decisions that must complete before work can continue:

- unsaved changes;
- destructive confirmation;
- replacement during restore;
- migration or corruption failure;
- closing during a non-cancellable commit;
- existing-process launch failure.

### Empty and loading states

Empty, loading, unavailable, and error states replace blank panes. Loading state identifies the operation in progress and exposes cancellation only when cancellation is safe.

---

## Responsiveness and Cancellation

File reads, parsing, matching, metadata retrieval, validation, migration, backup, restore, and persistence do not block the UI thread.

Long-running operations provide progress when meaningful. Indeterminate progress is used when total work cannot be known honestly.

Cancellation:

- is available during safe pre-commit work;
- is disabled once an atomic commit enters its non-cancellable section;
- never reports success before rollback or completion is known.

Closing the application during a non-cancellable commit waits for completion and explains why immediate exit is unavailable.

---

## Keyboard Contract

Every v0.1 workflow is completable without a pointer.

Baseline behaviour includes:

- Tab and Shift+Tab move through interactive controls in logical order;
- arrow keys move within lists, menus, and tab sets according to platform conventions;
- Enter opens or activates the selected item;
- Escape cancels or closes the current transient layer when safe;
- Delete opens the applicable confirmation and never deletes immediately;
- visible focus is retained;
- focus returns to a predictable triggering or next logical control after a transient layer closes.

Global shortcuts include:

- `Ctrl+N` — Create Workspace;
- `Ctrl+O` — Open Workspace;
- `Ctrl+S` — Save the active valid edit;
- `Esc` — Cancel the active draft or transient layer, subject to unsaved-change confirmation.

Version 0.1 does not include a command palette or custom shortcut editor.

---

## Accessibility

The UI uses standard controls and platform accessibility semantics wherever possible.

Requirements include:

- accessible names, roles, states, and descriptions;
- UI Automation exposure on Windows;
- headings and landmarks for major regions where supported;
- live announcements for status changes;
- visible keyboard focus;
- no information conveyed by colour alone;
- text and control contrast appropriate to the active theme;
- Windows high-contrast compatibility;
- Windows text scaling without clipped controls or unreachable primary actions;
- decorative elements excluded from the accessibility tree.

Custom controls require an appropriate automation peer, full keyboard behaviour, and accessible state exposure.

Automated accessibility checks do not replace manual Windows Narrator testing.

---

## Theme

The application uses Avalonia's Fluent theme.

Theme choices are:

- System — default, follows the Windows application theme;
- Light;
- Dark.

Theme selection is application-level settings data and does not belong to a Workspace.

Windows High Contrast takes precedence over System, Light, or Dark while it is active. The application retains the user's stored theme choice without applying it, then restores that choice when High Contrast is turned off. If High Contrast is active on first launch, the application uses the Windows High Contrast presentation and retains System as the default stored choice.

Version 0.1 does not provide custom skins or user-authored colour themes.

---

## Window Size and Placement

The main window opens at 1280 × 800 logical pixels when no valid saved placement exists.

The minimum size is 1024 × 640 logical pixels. The three-region Workspace shell remains visible at that minimum; it does not switch to a drawer, overlay, or list-then-detail navigation model.

The minimum supported display is 1280 × 720 at 100% Windows scaling. At increased scaling, the display must provide enough effective space for the 1024 × 640 logical minimum.

At launch and after a display or scaling change, the application compares the available Windows work area with its logical minimum. If the work area cannot contain the minimum window, the application shows a compatibility screen before Home or Workspace content. The screen explains that the current display or scaling configuration is unsupported and offers Exit or Continue Anyway. Continuing opens the window maximised at its minimum logical layout and keeps a compatibility warning visible for that session; clipping by the Windows work area may occur and complete workflow accessibility is not guaranteed. This exception does not weaken the no-clipping and reachable-action requirements on supported configurations.

The application remembers window size, position, and maximised state separately for installed and portable modes.

If display configuration changes or a remembered window is off-screen, the application restores it fully visible on the primary display.

At constrained sizes or increased scaling:

- panes scroll independently where needed;
- primary actions remain reachable without horizontal scrolling;
- controls and text do not overlap or clip;
- selection and focus remain visible.

---

## Settings Scope

Application settings include:

- theme choice;
- window placement;
- recent Workspace references;
- installed or portable mode-specific preferences;
- future non-Workspace presentation preferences.

Workspace-owned settings include only choices that change the Workspace's knowledge or target context.

The UI must not imply that an application preference is synchronised through Workspace JSON export.

---

## Privacy

The UI follows RFC-0003 and RFC-0004 privacy rules.

It does not expose full local paths unnecessarily, include Workspace content in diagnostics by default, fetch remote content implicitly, or render imported Markdown as trusted executable content.

Optional diagnostic packages disclose their contents before creation.

## Verification Criteria

### ViewModel tests

Tests cover:

- Home and Workspace shell navigation;
- Library and Membership selection;
- two-tab ownership boundaries;
- separate shared and Collection-local drafts;
- Edit, Save, Cancel, and unsaved-change outcomes;
- validation and Save availability;
- retained drafts after failure;
- success and error announcements;
- cancellable and non-cancellable operation state.

### Shell tests

Tests demonstrate that:

- Home appears before any Workspace file is accessed;
- missing recent files remain recoverable;
- only current-milestone destinations appear;
- Collection selection defaults to Collection Details;
- Library selection cannot expose Membership editing;
- shared and Membership drafts cannot save through the wrong command;
- returning Home resolves unsaved work safely;
- a second process exits before loading settings or Workspace data.

### Import UI tests

Using RFC-0004 fixtures, tests demonstrate that:

- the full preview remains visible;
- Needs Attention contains every blocking item;
- Resolve Next advances predictably;
- counts and table state update after resolution;
- Apply remains disabled until validation succeeds;
- confirmation mirrors the final domain plan;
- cancellation changes no Workspace data;
- application failure retains safe preview state and reports no commit.

### Destructive-action tests

Tests demonstrate that:

- Membership removal cannot call library deletion;
- library deletion remains disabled until complete impact data loads;
- every affected Collection appears;
- deleted and unresolved Relationship outcomes are distinguished;
- failure leaves data unchanged and reports the result.

### Accessibility tests

Automated and manual tests cover:

- keyboard-only completion of every v0.1 workflow;
- logical focus order and visible focus;
- accessible names, roles, states, headings, and announcements;
- list and tab keyboard navigation;
- System, Light, Dark, and Windows high-contrast presentation;
- Windows High Contrast overriding and then restoring each stored theme choice;
- representative Windows text-scaling settings;
- the compatibility screen and persistent session warning when the available work area is below the logical minimum;
- Windows Narrator operation.

### Packaging tests

Installer and portable builds are tested on clean Windows 10-or-later x64 environments.

Tests verify:

- no separately installed .NET or Avalonia runtime is required;
- settings use the accepted installed and portable locations;
- the single-instance rule works across both distribution modes;
- uninstall preserves or removes managed data according to the accepted option;
- user-selected external Workspace files remain untouched.

### Visual structure tests

Visual regression tests cover:

- Home;
- Workspace shell;
- both detail tabs;
- import preview and Needs Attention;
- destructive dialogs;
- major loading, empty, unavailable, and error states;
- supported themes and representative window sizes.

Tests detect clipping, overlap, missing states, and structural regressions rather than requiring pixel-identical platform rendering.

## Alternatives Considered

### WPF

WPF is mature, Windows-native, and compatible with self-contained .NET deployment. It would minimise initial platform abstraction but require replacing the presentation layer for future non-Windows desktop support. Rejected in favour of Avalonia while keeping v0.1 Windows-only.

### Tauri

Tauri offers flexible web-based UI composition. Fully offline Windows distribution may require bundling WebView2 and adds a separate web-runtime boundary. Rejected in favour of one .NET application stack.

### Dashboard-first navigation

A dashboard provides prominent summaries and friendly orientation. It adds a navigation step before routine Collection work and encourages premature summary features. Rejected in favour of Home followed by a direct Workspace shell.

### Document tabs

Document tabs support flexible multitasking but complicate context, unsaved-state management, and ownership visibility. Rejected for the single-window v0.1 workflow.

### Stacked or side-by-side ownership details

Stacked sections keep both ownership contexts visible but create a long detail pane. Side-by-side sections make ownership obvious but become cramped. Rejected in favour of two explicit tabs with separate drafts.

### Linear import wizard

A strict stepper makes progress obvious but hides earlier context during resolution. A single unified table becomes dense. Rejected in favour of a persistent preview with a guided Needs Attention queue.

### Responsive list-then-detail shell

Replacing the list with detail at narrow sizes uses space efficiently but changes navigation behaviour. A compressed split view reduces readability. Rejected in favour of a fixed minimum three-region shell for v0.1.

### Autosave

Per-field autosave reduces explicit actions but can persist partial multi-field edits and obscure whether shared or Collection-local data changed. Rejected in favour of explicit Edit, Save, and Cancel.

## Deferred Decisions

This RFC does not decide:

- exact .NET SDK, Avalonia, and dependency versions;
- the concrete .NET SQLite data-access library;
- installer technology and code signing;
- automatic update mechanism;
- final branding, iconography, typography, and spacing tokens;
- detailed v0.2 and later screens;
- undo and redo history beyond cancelling an active draft;
- command palette or configurable shortcuts;
- multi-window or detachable-pane support;
- ARM64 and non-Windows packages;
- telemetry or crash-reporting services;
- single-Collection export UI.

The Implementation Partner must propose the .NET SQLite data-access library in the storage implementation plan, demonstrate that it preserves RFC-0003's storage boundary and self-contained packaging requirements, and obtain Product Owner approval before storage code lands.

Exact dependency versions, window test matrix, and implementation-level design tokens must also be pinned in the implementation plan before application code lands.

---

## Consequences

- v0.0.2 has a concrete initial presentation architecture and technology choice.
- v0.1 implementation can begin only after RFC-0005 is accepted and the architecture gate is closed.
- Shared Mod and Collection Membership editing remain visibly and operationally separate.
- Import UI directly reflects RFC-0004's preview and resolution contract.
- Accessibility, keyboard operation, and packaging are release requirements.
- The fixed three-region shell establishes minimum window and display requirements.
- Future roadmap features enter the shell only when their architecture and milestone are approved.
- Acceptance requires README and roadmap updates that record the C#/.NET, Avalonia, and MVVM technology choice; the Windows 10-or-later x64 platform baseline; and the minimum supported display requirements.
