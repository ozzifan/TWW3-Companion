# RFC-0004: Import Architecture

**Status:** Proposed
**Author:** Project contributors
**Date:** 2026-07-16

---

## Summary

TWW3 Companion imports existing informal Markdown notes and Steam Workshop ID lists through deterministic source adapters. Every adapter produces a common candidate representation, and no adapter writes directly to the Workspace.

Imports are staged through normalisation, identity matching, an editable preview, required user resolutions, domain validation, and one atomic additive transaction. Exact Source References may match automatically; names and aliases only suggest possible matches.

The v0.1 importer does not infer relationships from prose, delete by omission, require network access, or create unresolved half-records in the Mod Library.

---

## Problem

Players already keep mod knowledge in informal notes rather than one standard template. Useful inputs include:

- Markdown headings and bullet lists;
- linked mod names;
- bare Steam Workshop URLs or IDs;
- numbered load-order notes;
- prose attached to a listed Mod;
- names for personal, local, or edited Mods with no publishing identity.

Requiring users to rewrite those notes into a project-specific grammar would undermine the purpose of import. Broad heuristic interpretation, however, risks false identity matches, invented relationships, silent data loss, and accidental overwrite of user-authored knowledge.

The import architecture must accept varied input while keeping every interpretation reviewable and every applied change safe.

---

## Scope

RFC-0004 defines inbound import for:

- informal Markdown documents;
- pasted or file-based Steam Workshop ID lists.

It defines parsing boundaries, candidate data, identity matching, preview and conflict behaviour, atomic application, optional metadata enrichment, privacy, errors, and verification.

It does not define:

- lossless Workspace JSON restore, which is governed by [RFC-0003](RFC-0003.md);
- single-Collection snapshot export;
- Markdown or ID-list export;
- mod package inspection;
- relationship inference from prose;
- replacement or synchronisation of a Collection.

---

## Chosen Approach

Import uses deterministic staged adapters:

```text
Input
→ source adapter
→ import candidates
→ normalisation
→ exact identity matching
→ suggested name matches
→ editable preview
→ required resolutions
→ domain validation
→ one atomic transaction
```

Markdown and Workshop-ID adapters share every stage after candidate creation.

This separation keeps parsing rules independent from domain mutation, makes matching explainable, and permits new source adapters without duplicating conflict and transaction rules.

---

## Components

### Source adapter

Reads one external representation and emits candidates, source locations, and parse diagnostics.

An adapter:

- does not read the Workspace;
- does not perform identity matching;
- does not create domain records;
- does not access persistence;
- does not make network requests.

### Candidate model

Represents proposed information without claiming that a Mod or Membership exists.

A candidate may contain:

- source location;
- proposed display name;
- one or more Source References;
- one proposed Collection category value;
- source order position;
- imported note segments;
- source document name and import time;
- adapter diagnostics.

Candidate identities are temporary to the import session and are not domain UUIDs.

### Normaliser

Canonicalises supported structural values without changing meaningful user prose.

It:

- trims structural Markdown and surrounding whitespace;
- normalises line endings for parsing;
- canonicalises recognised Steam Workshop URLs and numeric IDs;
- identifies exact duplicates within the input;
- preserves original line references and note text.

### Matcher

Compares normalised candidates with a read-only Workspace snapshot.

It:

- automatically matches an exact Source Reference;
- identifies exact existing Collection Memberships;
- suggests possible display-name or alias matches;
- never merges or creates records.

### Preview builder

Compares candidates with the target Collection and Mod Library, then describes proposed effects and required choices.

### Resolution session

Records link, create, skip, and conflict choices for the current import. It is temporary and is discarded when the user cancels or closes the import.

### Import planner

Converts a fully resolved preview into intention-revealing domain operations and validates the complete proposed change.

### Atomic applier

Applies the validated plan through RFC-0003's single SQLite transaction boundary.

Only the atomic applier may request persistence.

---

## Supported Inputs

### Informal Markdown

The Markdown adapter recognises candidates beginning with:

- unordered list items;
- numbered list items;
- bare supported Workshop URLs;
- bare numeric Workshop IDs;
- Markdown links whose destination is a supported Workshop URL.

A linked title or text adjacent to a recognised identity may propose a display name.

The importer supports UTF-8 with or without a byte-order mark and UTF-16 when identified by a byte-order mark. Inputs in other or ambiguous encodings fail with an actionable error rather than being decoded with replacement characters.

### Workshop ID lists

The Workshop-ID adapter accepts one Workshop ID or supported Workshop URL per logical line, with optional adjacent display text.

IDs are treated as strings of decimal digits after normalisation. The importer does not interpret them as database row numbers or internal identities.

### Pasted text and files

Pasted text and selected files use the same adapter behaviour after decoding. The importer does not retain clipboard contents or a full source path after the import session.

---

## Markdown Association Rules

### Headings and categories

The nearest preceding Markdown heading proposes one Collection category value for following candidates.

For nested headings:

- the nearest heading becomes the proposed category;
- higher headings remain visible as preview context;
- the adapter does not infer or create category hierarchy from heading depth.

This is an import interpretation rule, not a domain-wide decision that Categories must remain flat. A future Category design may support hierarchy without changing what v0.1 can safely infer from informal Markdown.

Category proposals remain editable in the preview.

### Category normalisation

Heading-derived category values are trimmed and runs of whitespace are collapsed before comparison.

The preview compares a proposed category case-insensitively with categories already used by the target Collection:

- one match reuses the existing category and preserves its stored display spelling;
- no match proposes a new category using the normalised heading's original letter case;
- more than one case-insensitive match is a conflict that requires the user to choose an existing category or edit the proposal.

Case differences alone do not create a new category automatically. Category comparison does not rename, merge, or otherwise mutate existing categories.

### Notes

Indented text and immediately following non-entry prose attach to the preceding candidate until another candidate or heading begins.

Attached text is preserved as an imported note segment. The parser removes structural indentation but does not rewrite wording, interpret Markdown emphasis, or infer meaning.

Unassociated prose remains visible as a diagnostic with its source lines.

### Ordering

Source order proposes documented ordering information on Collection Memberships.

Ordering is knowledge captured from the input. It is not described as a solved, validated, or enforced load order.

### Conservative interpretation

The parser does not infer:

- Dependencies;
- Compatibility Claims;
- patches or workarounds;
- load-before or load-after rules;
- game compatibility observations;
- local filesystem references.

Phrases such as “requires X,” “conflicts with Y,” or “load after Z” remain note text.

---

## Duplicate Input

Candidates with the same normalised Source Reference collapse into one candidate.

The combined candidate:

- retains every source location;
- retains distinct attached note segments;
- reports the duplicate as an informational diagnostic;
- reports conflicting proposed names or categories for preview resolution.

Exact duplicate note segments are retained once.

Name-only candidates do not collapse solely because their normalised names are equal.

---

## Identity Matching

### Exact Source References

A candidate whose normalised source type and external identifier match one library Mod selects that Mod automatically.

If the Workspace already violates Source Reference uniqueness, the import stops with an error. The importer does not choose between invalid matches.

### Names and aliases

Display names and aliases are not stable identities.

Name similarity may suggest existing Mods, but the user must explicitly choose:

- link to an existing Mod;
- create a new source-neutral Mod;
- skip the candidate.

The importer never automatically merges by name or alias.

### Source-neutral Mods

Candidates without a Source Reference are supported for personal, local, and edited Mods.

Before application, every name-only candidate must be linked, created, or skipped. Import candidates are not stored as unresolved or incomplete Mods.

Creating a source-neutral Mod assigns a stable domain UUID during application. A Source Reference may be attached later through a separate domain operation.

### Required display names

A valid Workshop ID without an adjacent or fetched title remains a candidate, but it blocks application until the user enters a non-empty display name or skips it.

The importer does not invent a placeholder title from the Workshop ID. Optional metadata may propose a title, but accepting it remains visible in the preview.

---

## Preview

Every candidate appears in a preview before any Workspace change.

The preview groups effects as:

- **Additions** — new Mods or Collection Memberships;
- **Enrichments** — blank fields to fill or note segments to append;
- **Existing** — exact identity and membership matches requiring no change;
- **Suggested matches** — name or alias suggestions requiring a choice;
- **Conflicts** — different non-empty values requiring resolution;
- **Warnings** — malformed, ambiguous, or unrecognised input;
- **Skipped** — candidates excluded by the user.

Each preview item retains its source document name and source line or line range.

The preview shows:

- current value where one exists;
- imported value;
- proposed result;
- validation severity;
- available choices;
- whether the item blocks application.

---

## Resolution Rules

### Required identity choices

Every suggested or name-only identity requires link, create, or skip before application.

Every new Mod requires a non-empty display name. An identity-only Workshop candidate requires user-entered or explicitly accepted imported metadata before application.

### Blank values

Imported values may fill blank fields after appearing in the preview.

### Equal values

Semantically equal values are informational and require no choice.

### Scalar conflicts

A different imported scalar value never overwrites a non-empty user-authored value automatically.

The preview offers:

- keep existing;
- use imported;
- skip the candidate where skipping remains valid.

### Notes

Distinct imported note text appends by default rather than replacing existing notes.

Each appended segment remains distinct and records:

- import date;
- source document name;
- source line or line range.

The full local path is not stored by default.

Exact duplicate note segments do not append again. The preview shows the resulting combined notes before confirmation.

### Apply to similar

The user may apply one conflict choice to similar fields in the current import session.

The preview identifies every affected item, and the final summary displays the expanded choices. The rule is not stored as a silent global preference.

The exact grouping rule that determines which fields are “similar,” and whether this bulk action is included in v0.1, are deferred to the implementation plan. A bulk choice may not cross field types or resolution actions unless that behaviour is explicitly documented and tested.

---

## Additive-Only Collection Changes

Version 0.1 imports are additive-only.

An import may:

- create a confirmed source-neutral Mod;
- add a Source Reference to a newly created Mod;
- add a Mod to the target Collection;
- fill approved blank fields;
- append distinct imported note segments;
- set approved category and ordering proposals.

An import does not:

- remove an existing Collection Membership;
- delete a library Mod;
- clear a non-empty field;
- infer deletion from omission;
- replace or synchronise a Collection;
- move a Profile;
- delete Relationships or Evidence.

Replacement and synchronisation require a separate design with explicit deletion semantics.

---

## Diagnostics

Diagnostics use RFC-0002's Error, Warning, and Information severities.

### Error

An Error blocks application. Examples include:

- unreadable input;
- unsupported or ambiguous text encoding;
- resource limit exceeded;
- a Source Reference resolving to an invalid non-unique Workspace state;
- an unresolved required identity choice;
- an invalid proposed domain operation.

### Warning

A Warning remains reviewable but does not block after required choices are resolved. Examples include:

- malformed Workshop URL or ID;
- candidate without a stable Source Reference;
- ambiguous Markdown association;
- unrecognised line;
- optional metadata unavailable.

### Information

Information requires no intervention. Examples include:

- duplicate input collapsed;
- existing Collection Membership;
- identical field or note segment;
- candidate explicitly skipped.

Every diagnostic identifies the affected candidate and source location and proposes a remedy where one exists.

---

## Confirmation and Application

Application is available only when:

- all Errors are resolved;
- every required identity choice is recorded;
- every blocking conflict is resolved;
- the complete domain plan passes validation.

Before confirmation, the UI shows exact counts and the final list of domain operations.

Confirmation follows:

```text
Resolved preview
→ build domain operation plan
→ validate complete plan
→ explicit confirmation
→ begin SQLite transaction
→ apply every operation
→ commit
```

If validation or persistence fails, the transaction rolls back completely. The application reports that no changes were committed and returns to the resolved preview where safe.

Cancelling or closing an import discards the temporary session and changes nothing.

---

## Optional Workshop Metadata

Workshop metadata enrichment is optional, explicit, and separate from parsing and identity import.

Before a request, the UI identifies which Workshop IDs will be contacted. The user initiates the request.

Network access:

- is never required to import a valid Workshop ID;
- does not occur implicitly when a file is selected or previewed;
- is limited to the IDs disclosed for the current request;
- records source and observation time for accepted metadata.

Network failure, rate limiting, unavailable items, and partial responses produce warnings. They do not block identity-only import.

Retrieved titles may propose display names. They do not override user-authored names without explicit conflict resolution.

---

## Privacy and Input Safety

The importer reads only the selected file or pasted text.

It does not:

- follow arbitrary local links;
- open files referenced by Markdown;
- inspect or copy mod packages;
- execute HTML, scripts, or embedded content;
- fetch images or arbitrary URLs;
- write beside the source document;
- include imported prose or clipboard text in logs by default.

Markdown is parsed as untrusted data. Preview rendering must escape or sanitise content according to the chosen UI technology.

Import provenance stores the source document name, not its full local path, by default.

A user-created diagnostic package may include additional information only after identifying what will be included and obtaining explicit confirmation.

---

## Resource Limits

Parsing is bounded to prevent accidental or malicious resource exhaustion.

The implementation plan must define and test:

- maximum input bytes;
- maximum logical lines;
- maximum candidates;
- maximum text per candidate;
- maximum diagnostics retained and displayed.

When a limit is exceeded, parsing stops before preview, identifies the exceeded limit, and makes no Workspace change. Limits must be documented for users and may not silently truncate input.

---

## Session Lifetime

Import candidates, suggested matches, and resolution choices are temporary application state.

Version 0.1 does not persist resumable import sessions. Closing the application or starting a different import prompts before discarding unsaved choices. Confirmed cancellation discards the current session.

Source document content, clipboard text, and full paths are not persisted as session recovery data.

## Verification Criteria

### Adapter fixtures

Fixtures cover:

- headings and nested headings;
- unordered and numbered lists;
- bare Workshop IDs;
- supported Workshop URLs;
- Markdown-linked names;
- adjacent and indented notes;
- UTF-8, UTF-8 with byte-order mark, and byte-order-marked UTF-16;
- Windows and Unix line endings;
- duplicate IDs and duplicate note segments;
- malformed identities;
- unrecognised prose;
- resource-limit failures.

### Matching

Tests demonstrate that:

- exact normalised Source References match automatically;
- names and aliases only suggest;
- equal names do not collapse source-neutral candidates;
- name-only candidates block until link, create, or skip;
- re-imported name-only candidates are suggested again and are not automatically deduplicated;
- ID-only candidates block until they have a user-entered or explicitly accepted display name;
- matching never mutates the Workspace.

### Preview and resolution

Tests cover every preview group and conflict choice.

They demonstrate that:

- headings propose one editable category value without inferring hierarchy;
- category whitespace is normalised and case-insensitive matches reuse existing display spelling;
- `UI`, `ui`, and spacing variations of the same text do not automatically create separate categories;
- ambiguous existing case-insensitive category matches require resolution;
- source order proposes documented ordering;
- blank fields may be filled;
- scalar conflicts never overwrite silently;
- distinct notes append with provenance;
- exact note segments do not duplicate;
- apply-to-similar is limited to one session;
- cancelling changes nothing.

### Safety and privacy

Tests demonstrate that input cannot trigger:

- script or HTML execution;
- local-file traversal;
- package inspection;
- implicit network access;
- image fetching;
- logging of note or clipboard content;
- storage of full source paths by default.

### Atomic application

Tests use real temporary SQLite Workspaces and demonstrate:

- additive-only behaviour;
- no deletion by omission;
- equivalence between the confirmed operation list and persisted results;
- complete rollback after any application failure;
- re-importing identical Source-Reference-backed input does not duplicate Memberships or note segments;
- re-importing name-only input does not auto-match and may create a distinct Mod if the user explicitly chooses create again.

### Optional metadata

Tests simulate:

- offline operation;
- rate limiting;
- missing Workshop items;
- partial responses;
- conflicting fetched titles.

In every case, valid Workshop identities remain importable without metadata.

Golden fixtures under `examples/` and `tests/` document supported informal patterns and expected diagnostics without becoming a mandatory Markdown template.

## Alternatives Considered

### Strict project Markdown grammar

A mandatory template is predictable and simple to validate. It forces users to rewrite existing notes before import and conflicts with the goal of accepting practical source material. Rejected for v0.1.

### Combined smart importer

One pipeline could parse, match, and decide mutations together. It tangles format assumptions with domain rules, makes dry-run preview less trustworthy, and is harder to test stage by stage. Rejected in favour of staged adapters.

### AI-assisted semantic import

An AI service could interpret more prose and suggest relationships. It introduces nondeterminism, privacy and network concerns, and a risk of invented knowledge. It conflicts with conservative, offline-first v0.1 import. Deferred as a possible explicit future adapter, not a foundation.

### Automatic name matching

Automatically merging equal or similar names reduces user choices but can combine unrelated, forked, local, or edited Mods. Rejected because names are not stable identities.

### Replace or synchronise Collection

Treating omission as deletion can reproduce an external list exactly. It makes an informal partial document destructive and requires deletion semantics beyond additive import. Rejected for v0.1.

---

## Deferred Decisions

This RFC does not decide:

- exact parser libraries or implementation language;
- exact resource-limit values;
- name-similarity algorithm and suggestion threshold;
- apply-to-similar availability and field-grouping rules;
- detailed preview layout;
- saved mapping presets across import sessions;
- additional source adapters;
- relationship extraction from structured future formats;
- replace or synchronise behaviour;
- single-Collection snapshot export;
- Markdown, ID-list, or other sharing exports;
- optional AI-assisted import.

These require implementation planning or separate architecture approval where they affect user-visible behaviour.

---

## Consequences

- v0.1 can import existing informal notes without making Markdown a proprietary storage format.
- All source adapters share one explainable matching, preview, and atomic application pipeline.
- Source-neutral personal and edited Mods remain supported.
- Imports cannot remove knowledge by omission.
- Users must resolve ambiguous identities before application.
- Optional metadata improves convenience without making the network a core dependency.
- Example fixtures and diagnostics become part of the documented import contract.
- Single-Collection export remains a separate design requirement before scoped sharing is implemented.
